# Integration Guide

This is the priority cross-service doc for VITAL_Services. Read it before any change that touches request trust, identity, or event publishing.

## 1. Gateway is the only browser entry point

```
browser  ──►  BGH_API_GATEWAY  ──►  VITAL_Services
                  /vital/*           /vital/*
```

Browsers must never call VITAL_Services on `:8009` directly. The Gateway:

1. Verifies the session/JWT.
2. Performs silent refresh and may emit `X-New-Access-Token` to the browser.
3. Injects the trusted `X-Gateway-Secret` and `X-User-*` headers.
4. Forwards `/vital/*` to `http://vital-services-api:8009`.

## 2. Trust model

| Step | Header | Source of truth |
|---|---|---|
| Validate caller | `X-Gateway-Secret` | `.env.vital_services` `GATEWAY_SECRET` (must equal Gateway's value) |
| Identify user | `X-User-Id`, `X-User-Email`, `X-User-Roles` | Set by Gateway after JWT verification |
| Identify system context | `X-System-Context` | Set by Gateway (slug like `vital`, `uhse`) |
| Trace request | `X-Correlation-ID` | Generated by Gateway or upstream caller |

`src/middleware/trustedGateway.js` enforces this. Never read `X-User-*` outside this middleware.

## 3. Direct calls to UHSE_AUTH

When VITAL_Services needs to:

- enrich a user (full profile, system roles)
- validate a token out-of-band
- look up identity for a long-running job

…it calls `AUTH_INTERNAL_BASE_URL` directly (e.g. `http://auth-api:8000/internal/...`).

Pass `INTERNAL_API_KEY` (or the dedicated `AUTH_INTERNAL_SERVICE_KEY` when Auth requires per-caller keys) and propagate `X-Correlation-ID`.

**Never** loop these calls back through the Gateway.

## 4. Auth role aggregation contract

`UHSE_AUTH` periodically (or on-demand) calls `/internal/users/{user_id}/roles` on every system that participates in the `vital` slug. VITAL_Services owns this endpoint for `vital`.

- Response shape MUST be `{ "system": "vital", "user_id": "...", "roles": [...] }`.
- Empty roles is a valid response (no VITAL-specific roles for this user).
- Guard with `trustedGateway` OR with a dedicated internal-service key — coordinate with `UHSE_AUTH/.agents/rules/`.

## 5. Audit publishing

VITAL_Services publishes business events to RabbitMQ:

- URL: `RABBITMQ_URL` (default `amqp://guest:guest@bgh-rabbitmq:5672/`)
- Exchange: `AUDIT_EXCHANGE` = `audit.events`
- Envelope: follow `Documents/Audit_Logs_Integration_Guide_v2.md`
- Always include `X-Correlation-ID` in the event payload so audit traces line up with logs.

## 6. Correlation propagation

Every:

- inbound request reads `X-Correlation-ID` (or generates one)
- outbound HTTP call carries it
- audit envelope carries it
- log line carries it (`correlation_id` field)

This is non-negotiable — operators rely on it to trace a session across UHSE Frontend, Gateway, VITAL Services, Audit, and Auth.

## 7. What VITAL_Services does NOT own

- Login, logout, password, MFA — `UHSE_AUTH`.
- `X-Gateway-Secret` issuance / route classification — `BGH_API_GATEWAY`.
- Audit retrieval and storage — `BGH_AUDIT_LOGS`.
- Browser UX — `VITAL_WEB`.
